- The repeat-task save/toggle issue is a post-save scheduler-feedback bug rather than a persistence bug: `saveLoop(...)` can succeed while the follow-up runtime sync (`startLoop(...)` / `stopLoop(...)`) still rejects or returns `{ success: false }`, so bundling them into one optimistic success/failure path misleads the user about what actually happened.
- Splitting persistence from scheduler sync and only warning on `stopLoop(...).success === false` when the loop still looked scheduled is the smallest safe fix because it preserves existing loop-service semantics, keeps benign already-stopped disables quiet, and makes real partial failures visible without inventing new backend behavior.
- The new tool-approval issue is a boolean-contract mismatch rather than another thrown-error path: the renderer now handled rejected approval IPC calls, but it still treated resolved `{ success: false }` responses as success even though the main process uses that value to mean `approval is already gone`.
- Converting stale false-results into the existing error/reset path is the smallest safe fix because it preserves the current approval UI, avoids introducing a new cross-window state abstraction, and immediately prevents the user-facing `Processing...` dead-end on the active desktop approval surfaces.

- The floating-panel issue is a direct contract mismatch between Electron window-drag regions and this component's own implementation: `PanelDragBar` was not using native frameless dragging; it was using a custom pointer-driven drag lifecycle that needs `onMouseDown` / `mousemove` events to fire on the handle itself.
- For that implementation, forcing the visible handle into `WebkitAppRegion: "no-drag"` is the smallest safe fix because it preserves the existing IPC throttling, screen-constrained movement, and custom-position persistence without widening into main-process window-drag refactors.
- The onboarding issue is another explicit boolean/result-contract mismatch on a primary setup CTA: the config-save step can succeed while the later runtime-enable/start step reports a non-throwing failure, and treating every resolved promise as success makes the user-visible install state less honest than the backend state.
- Reusing the existing saved-but-not-started recovery UI is the smallest safe fix because it preserves the current onboarding layout, keeps successful installs unchanged, and only broadens the failure detection to match the main-process contract instead of inventing a new setup flow.
- The MCP bulk-toggle issue is a state-rollback correctness bug rather than just missing feedback: these handlers already warned when some per-server tool toggles failed, but their rollback path assumed every tool started opposite the requested end state and could therefore paint a false enabled/disabled result after a partial failure.
- Capturing each tool's original enabled state and restoring from that snapshot is the smallest safe fix because it preserves the existing optimistic UI and warning toasts while aligning the renderer's rollback behavior with the backend's real `success: false` contract.
- The remaining Settings → Agents issue was a backend persistence-truthfulness bug rather than another renderer feedback bug: the desktop editor already respected falsy/failed mutation results, but the main process could still swallow `saveProfiles()` write failures and return mutated in-memory profiles as if they were durably saved.
- Snapshotting and restoring in-memory profile state around `saveProfiles()` is the smallest safe fix because it preserves current TIPC/renderer contracts while preventing unsaved transient profile mutations from leaking into the running app after a failed write.
- The new repeat-task issue was a backend persistence-truthfulness bug rather than another scheduler-feedback gap: the desktop renderer already had a concrete `saveLoop(...)` step, but the main process always resolved `{ success: true }` even when the canonical `task.md` write failed.
- Rolling the loop-service state back and forwarding that real boolean result is the smallest safe fix because it keeps the existing desktop loop editor/scheduler flow intact while finally making “saved” mean the repeat task actually reached disk.
